# 1. Resumen Ejecutivo
El Sistema CompraVenta v6.0 es una solución empresarial diseñada para la gestión integral de establecimientos de compra, venta y empeño. Esta versión consolida los requisitos previos (v4 y v5) en una arquitectura robusta, escalable y con una experiencia de usuario (UX) optimizada para operaciones de alto tráfico en mostrador. El sistema garantiza integridad transaccional, auditoría completa y una interfaz moderna.

# 2. Problemas Detectados en el Sistema Actual
Tras un análisis profundo del código y la base de datos v4.0, se identificaron los siguientes puntos críticos:
- **Inconsistencia de Modelos:** Los documentos v5.0 proponen campos (como `registration_type` o `source_type`) que no existen en el código ni en la base de datos actual.
- **Riesgo de Stock:** Aunque existe una stored procedure para ventas, algunos flujos de empeño manejan el stock desde Java, lo que puede causar inconsistencias en caso de errores de red.
- **UX Lenta:** Los formularios actuales requieren múltiples clics y cambios de módulo para registrar a un cliente nuevo antes de una operación.
- **Seguridad Fragmentada:** La lógica de permisos está ahora centralizada en políticas RLS de Supabase, actuando como cortafuegos principal.
- **Falta de Entorno Local:** Se ha implementado `docker-compose.yml` para desarrollo offline con PostgreSQL 16.
- **Estado Actual:** El sistema se encuentra en versión 6.1 estable. Se ha completado la refactorización de nomenclatura (Profile -> Employee) y la estandarización de estados de cliente (ClienteStatus), sincronizando el 100% de la capa DAO con database_v6.sql.

# 3. Arquitectura Recomendada
Se propone una evolución hacia una arquitectura de 4 capas:
1.  **Capa de Presentación (UI):** Java Swing con **FlatLaf** (Look & Feel moderno). Uso de **MVP (Model-View-Presenter)** para separar la lógica de vista.
2.  **Capa de Servicio:** Lógica de negocio centralizada, orquestación de transacciones y transformaciones de DTO.
3.  **Capa de Persistencia (Repository/DAO):** Abstracción del acceso a datos. Uso de **ConnectionPool (HikariCP)**.
4.  **Base de Datos:** PostgreSQL 16 con lógica pesada en Stored Procedures para asegurar la integridad atómica.

# 4. Módulos del Sistema

### 4.1 Login & Seguridad
- Autenticación vía Supabase Auth.
- Gestión de sesiones basada en la entidad `Employee` (Tabla `employees`).
- Roles: `Admin` (Control total) y `Empleado` (Operativo con restricciones).

### 4.2 Dashboard
- Métricas en tiempo real: Empeños activos/vencidos, valor de inventario, ventas del día.
- Alertas visuales para artículos próximos a vencer.
- Métricas de clientes sincronizadas (`incomplete_clients`).

### 4.3 Clientes (v6)
- **Registro Rápido:** Creación de cliente con solo Nombre y Teléfono desde cualquier módulo.
- **Perfiles Completos:** Gestión de DNI/Cédula, Dirección y Correo.
- **Historial:** Trazabilidad total de todas las operaciones del cliente.

### 4.4 Inventario & Compras
- **Source Type:** Identificación de origen (Empeño, Compra, Ajuste).
- **Control de Estado:** Nuevo, Bueno, Regular, Dañado.
- **Módulo de Compras:** Registro de adquisiciones directas del negocio a clientes/proveedores.

### 4.5 Empeños (Pawns)
- **Flujo Unificado:** Registro de Cliente + Artículo + Empeño en una sola transacción.
- **Gestión de Cuotas:** Seguimiento automático de pagos y morosidad.
- **Transición Automática:** Cambio a estado `Vencido` o `Perdido` basado en reglas de negocio.

### 4.6 Ventas & Facturación
- **Carrito de Compras:** Venta multiartículo rápida.
- **Ventas Anónimas:** Soporte para clientes no registrados.
- **Generación PDF:** Facturación automática con secuencias diarias seguras.

### 4.7 Auditoría & Reportes
- Registro automático de cada cambio en tablas críticas (`audit_log`).
- Reportes de cierre de caja, inventario valorizado y rentabilidad.

# 5. Historias de Usuario (Principales v6)
- **HU-01:** Como cajero, quiero registrar una venta en menos de 20 segundos para agilizar la atención.
- **HU-02:** Como administrador, quiero ver el valor total de los empeños activos para conocer el riesgo financiero.
- **HU-03:** Como empleado, quiero registrar un cliente rápido durante un empeño para no perder tiempo.

# 6. Reglas de Negocio Implementadas (v6.1)
- **RN-01:** Un empeño pasa a estado `Vencido` automáticamente si `return_date < CURRENT_DATE` (procesado en BD).
- **RN-02:** El stock de un artículo se reduce de forma atómica mediante la SP `register_sale`. No se puede vender si `amount < stock_solicitado`.
- **RN-03:** Los pagos de cuotas reinician el contador de cuotas perdidas (`installments_missed = 0`) y aumentan `installments_paid`.
- **RN-04:** La factura secuencial se genera mediante `next_invoice_number()` garantizando unicidad diaria.

# 7. Diseño UI/UX Recomendado
- **Look & Feel:** Uso de FlatLaf Dark/Light.
- **Navegación:** Menú lateral persistente.
- **Tablas:** Modernas con filtrado instantáneo y paginación.
- **Micro-interacciones:** Spinners de carga asíncronos para evitar bloqueos del sistema.

# 8. Seguridad & Escalabilidad
- **RLS:** Row Level Security en PostgreSQL como cortafuegos principal.
- **Docker:** Entorno contenedorizado para fácil despliegue y réplicas locales.
- **Variables de Entorno:** Separación estricta de configuraciones Dev/Prod.

# 9. Roadmap de Mejoras
1.  **Fase 1:** Migración de Base de Datos a v6.0 y configuración de Docker.
2.  **Fase 2:** Implementación de Capa de Servicio con DTOs y Validaciones Centralizadas.
3.  **Fase 3:** Rediseño UI con FlatLaf y Módulos de Operación Rápida.
4.  **Fase 4:** Módulo de Reportes Avanzados y Sincronización Automática Supabase.
