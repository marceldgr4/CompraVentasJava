## 1. Resumen de Análisis (V5 vs Realidad)
Tras auditar el código fuente y el esquema de base de datos actual, se concluye que el sistema se encuentra en una **fase de transición crítica**. Mientras que la base de datos ya cuenta con el diseño para la versión 6.0, el código Java aún no implementa todas las extensiones propuestas en la v5.0.

### 🟡 ¿Qué está LISTO? (Ready)
1.  **Infraestructura de Datos (SQL)**: El archivo `database_v6.sql` es excelente y soporta el 100% de los requisitos de V5 y V6 (Triggers, Auditoría, Stored Procedures, Enums).
2.  **Módulo de Empeño (Base)**: El modelo `Pawn.java` ya tiene los campos de cuotas (`installment_count`, `installments_paid`) y peso (`weight_grams`).
3.  **Seguridad**: `SessionManager` y `AuthService` están operativos con Supabase.
4.  **Venta Atómica**: `SaleService` ya consume la Stored Procedure `register_sale`, garantizando integridad de stock. Soporta ventas anónimas (`p_nombre_anon`).
5.  **Carrito de Compras**: La interfaz `SaleDialog` ya implementa un carrito multi-artículo funcional.
6.  **Modelos v6**: Los POJOs Java (`Article`, `Cliente`, `Pawn`, `Employee`) ya están sincronizados con `database_v6.sql` (Enums, nomenclatura Employees, ClienteStatus).

---

### 🔴 ¿Qué FALTA por desarrollar?
1.  **Módulo de Compras (Purchases)**: No existe `PurchaseService`, `PurchaseDao` ni la lógica para que el negocio compre artículos (HU-28).
2.  **Facturación PDF**: No hay implementación de iText/PDFBox para generar los comprobantes (HU-24).
3.  **Registro Rápido de Clientes**: Falta el diálogo modal simplificado que permita crear un cliente con solo nombre y teléfono sin salir de una operación (HU-25).
4.  **Pantalla Única de Empeño**: No se ha desarrollado la vista unificada (HU-26) que combina creación de cliente + artículo + empeño.
5.  **Venta Rápida (UX)**: Falta la pantalla optimizada para mostrador con buscador instantáneo y atajos de teclado (HU-27). (En progreso).
6.  **Automatización de Expiración**: Falta el `SwingWorker` que ejecute la limpieza de empeños vencidos al iniciar la aplicación (HU-15).

---

### 🟠 ¿Qué toca MODIFICAR o cambiar?
1.  **Lógica de Validación**:
    *   `ClienteService`: Actualmente obliga a ingresar Apellido, Email y Teléfono. Debe relajarse para el tipo `RAPIDO` (solo Nombre es obligatorio).
2.  **Stored Procedures en Java**:
    *   Actualizar `SaleService` para que soporte el parámetro opcional `p_nombre_anon` de la función `register_sale`.
3.  **DAOs**: Los DAOs principales (`ArticleDao`, `ClienteDao`, `PawnDao`) ya han sido actualizados para soportar las columnas de v6.sql.

## 2. Recomendaciones de Nivel AVANZADO (Mejoras v6.0)

Para alcanzar el nivel de "Proyecto Profesional" buscado en la v6.0, se sugieren estas mejoras técnicas:

### 🚀 Mejoras de Backend y Datos
*   **Capa de DTOs**: Separar los modelos de base de datos de los objetos de la UI para evitar pasar entidades incompletas.
*   **Pool de Conexiones**: Asegurar el uso de **HikariCP** (ya referenciado en v6) para evitar latencias con Supabase.
*   **Logging Estructurado**: Reemplazar los `System.out.println` o `e.printStackTrace()` por un logger (Log4j2 o SLF4J) hacia un archivo de auditoría local además de la tabla `audit_log`.

### 🎨 Mejoras de UI/UX (Premium)
*   **Look & Feel Moderno**: Integrar **FlatLaf** (IntelliJ Theme) para que la aplicación deje de verse como un Swing estándar de Windows y tenga un aspecto oscuro/moderno.
*   **Buscadores con Debounce**: Implementar un retardo de 300ms en los campos de búsqueda para no saturar la base de datos con cada pulsación de tecla.
*   **Micro-animaciones**: Usar un `GlassPane` para mostrar un spinner de carga translúcido durante las peticiones a Supabase, evitando que la UI parezca "congelada".
