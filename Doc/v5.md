# 1. Resumen Ejecutivo y Contexto del Sistema

## 1.1 Descripción General

CompraVenta es una aplicación de escritorio (Java 21 + Swing) orientada a la gestión integral de operaciones de compra, venta y empeño de artículos de segunda mano. Persiste su información en Supabase (PostgreSQL 16) y delega la autenticación a Supabase Auth. El sistema soporta dos roles: Administrador y Empleado, con permisos diferenciados implementados mediante Row Level Security en base de datos y validación en capa de servicio.

  (SELECT COUNT(*) FROM articles WHERE amount > 0)                         AS total_articles,
  (SELECT COUNT(*) FROM clientes WHERE status = 'Activo')                  AS total_clientes,
  COALESCE(SUM(p.price * p.amount) FILTER (WHERE p.status = 'ACTIVO'), 0) AS total_active_value,

## 5.3 Normalización y Justificación 3FN v5.0

El modelo sigue cumpliendo 3FN con las modificaciones de v5.0:

- **clientes v5.0:** La columna `registration_type` depende exclusivamente del `id` del cliente (no de otros atributos no-clave). `last_name` y `email` son nullable pero no crean dependencias parciales. La CHECK constraint garantiza consistencia.
- **articles v5.0:** `source_type` y `purchase_price` son atributos propios del artículo; no dependen de category ni de cliente. `purchase_price` es el costo de adquisición del negocio, independiente del `price` (precio de venta).
- **sales v5.0:** `cliente_nombre_temp` es un snapshot de conveniencia para ventas anónimas. Al ser nullable y usarse solo cuando `cliente_id IS NULL`, no introduce dependencias transitivas.
- **purchases (nueva):** PK = id. `purchase_price`, `sale_price_suggested`, `condition_at_purchase` y `notes` son atributos propios de la compra. La referencia a `article_id` es FK, no dependencia funcional. Cumple 3FN.

## 5.5 Transacciones y Atomicidad v5.0

**Empeños (sin cambios):** La reducción de stock se realiza en transacción JDBC (setAutoCommit(false)) que incluye tanto el INSERT en pawns como el UPDATE en articles.amount.

**Empeño en pantalla única (NUEVO v5.0):** La transacción JDBC incluye hasta cuatro pasos: [INSERT clientes si cliente nuevo] + [INSERT articles si artículo nuevo] + INSERT pawns + UPDATE articles.amount. Si cualquier paso falla, se ejecuta rollback completo de todos los inserts.

```java
// PawnSingleScreenService.java (ejemplo de estructura de transacción)
public PawnResult registerPawnSingleScreen(PawnSingleScreenRequest request) {
    Connection conn = connectionPool.getConnection();
    try {
        conn.setAutoCommit(false);
        Integer clienteId = resolveCliente(conn, request.getClienteData());
        Integer articleId = resolveArticle(conn, request.getArticleData());
        PawnRecord pawn = pawnDAO.insert(conn, buildPawnRecord(request, clienteId, articleId));
        articleDAO.decrementStock(conn, articleId, request.getAmount());
        auditDAO.log(conn, "pawns", "INSERT", pawn.id(), null, pawn);
        conn.commit();
        return PawnResult.success(pawn);
    } catch (Exception e) {
        conn.rollback();
        throw new ServiceException("Error al registrar empeño: " + e.getMessage(), e);
    } finally {
        conn.setAutoCommit(true);
        conn.close();
    }
}
```

**Compras (NUEVO v5.0):** La transacción JDBC incluye: [INSERT clientes si cliente nuevo] + INSERT articles (source_type=COMPRA) + INSERT purchases. Todo atómico o rollback completo.

**Ventas:** Se usa la stored procedure register_sale(p_profile_id, p_cliente_id NULLABLE, p_cliente_nombre_temp NULLABLE, p_items JSONB) que ejecuta en una única transacción PostgreSQL con lock pesimista FOR UPDATE. El ERRCODE CV001 indica stock insuficiente.

**Pagos de cuotas (sin cambios):** El trigger update_pawn_status_on_payment() actualiza atómicamente installments_paid y el estado del empeño.

# 6. Propuesta de Rediseño de Formularios (UX/UI v5.0)

## 6.1 Principios de Diseño Aplicados

Los formularios actualizados en v5.0 aplican los siguientes principios para reducir el tiempo de operación en mostrador:

- **Formularios dinámicos:** Los campos se muestran u ocultan según el modo seleccionado (cliente existente / cliente nuevo). No se muestran campos innecesarios.
- **Autocompletado en todas las búsquedas:** Los buscadores de cliente y artículo responden en tiempo real con debounce configurable.
- **Foco automático:** Al abrir cualquier formulario de operación, el cursor se posiciona automáticamente en el campo de mayor frecuencia de uso (campo de búsqueda).
- **Validaciones no bloqueantes donde sea posible:** Advertencias de precio (compra >= venta) no bloquean la operación, solo requieren confirmación.
- **Confirmación única:** Un solo botón de acción principal completa la operación. No hay pasos intermedios innecesarios.

# 10. Recomendaciones de Rendimiento v5.0

## 10.1 Consultas Críticas

Las siguientes consultas deben estar optimizadas con EXPLAIN ANALYZE antes de liberar a producción:

- **Búsqueda de cliente en venta rápida:** La query sobre `clientes` con índices GIN sobre nombre, apellido y teléfono debe retornar en < 200ms para hasta 50.000 registros.
- **Búsqueda de artículo disponible:** La query sobre `articles WHERE amount > 0` usa el índice parcial `idx_articles_available`. Verificar que el query planner lo use con EXPLAIN.
- **Carga del listado de empeños activos:** Siempre debe filtrar por `status = 'ACTIVO'` para usar `idx_pawns_activos` (índice parcial).

## 10.2 Diseño del VentaCarritoPresenter

El carrito de ventas no debe realizar ninguna consulta a la BD hasta que el empleado presione "Confirmar Venta". Todo el estado previo (artículos agregados, cantidades, totales) se mantiene en memoria en el presenter:

```java
public class VentaCarritoPresenter {
    private final List<CarritoItem> items = new ArrayList<>();
    private ClienteDTO clienteSeleccionado; // nullable para venta anónima
    private String clienteNombreTemp;       // para venta con nombre libre

    public void agregarArticulo(ArticleDTO articulo) { ... }
    public void modificarCantidad(int articleId, int nuevaCantidad) { ... }
    public BigDecimal calcularTotal() { ... }
    public BigDecimal calcularCambio(BigDecimal montoRecibido) { ... }
    public void confirmarVenta(VentaService service) { ... } // única llamada a BD
}
```

## 10.3 Transacción de Empeño en Pantalla Única

Para minimizar el tiempo de lock en BD durante la transacción de pantalla única, el orden de operaciones debe ser:

1. INSERT en `clientes` (si nuevo) - operación más liviana, sin locks sobre inventario.
2. INSERT en `articles` (si nuevo) - sin reducción de stock todavía.
3. INSERT en `pawns` - referencia al artículo y cliente ya existentes.
4. `SELECT amount FOR UPDATE` en `articles` - el lock más corto posible.
5. `UPDATE articles SET amount = amount - ?` - validar stock > 0 antes.
6. INSERT en `audit_log`.
7. COMMIT.

Este orden minimiza el tiempo de lock sobre la tabla `articles` al final, reduciendo la posibilidad de bloqueos bajo carga concurrente.
